notePrompt × showTimingdesign options

The question: when a question has a free-text notePrompt AND the group has the "now or later?" timing flow on, how should the card behave? Click Yes in each variant to see the resulting flow. Pros & cons on the right.

"Threesome with an additional partner who has a vulva" — a real notePrompt question from questions.yml.

A. Skip timing on notePrompt questions

Yes/willing on a notePrompt question commits with timing: null and stays in Layout B for the note. The now/later sub-question is suppressed.

Pros

  • Removes the UI papercut where the rating buttons reappeared after picking "Now."
  • One signal at a time — never two open questions on the same card.
  • Fewer clicks on questions that benefit most from a note.

Cons

  • Silently breaks the data contract. showTiming stops applying to a subset of questions, with no UI for the user to set timing.
  • Timing was richest on exactly the questions that have notePrompt (threesomes, signal words, sexting boundaries).
  • Mixed journals: some yes answers carry timing, some don't, no route to backfill.
  • The matching algorithm loses signal where it mattered most.

Verdict: currently shipped, but the data-loss cost looks larger than the UI win that motivated it.

B-inline: keep timing, render alongside the note

Yes/willing reveals the note section AND timing buttons in the same Layout B card. Single screen, two signals, one Next.

Pros

  • Preserves both signals — timing data isn't lost on notePrompt questions.
  • Resolves the original papercut: rating buttons aren't replaced by anything; timing lives in the note section, not in place of it.
  • Single card, single Next — the user fills as much as they want and advances once.
  • No data-contract violation — showTiming applies uniformly.

Cons

  • Layout B grows in height (extra row of buttons + label).
  • Two prompts ask "answer me" at once — slightly more cognitive load than the sequential variant.
  • Need to decide whether timing is required to advance, or optional like the note. (Recommended: optional, matching the note's optionality.)

Verdict: respects both signals and the original UI motivation. My recommendation.

B-sequential: rate → note → timing

Three explicit phases. Yes opens the note. Filling and pressing Next reveals the timing sub-question. Picking timing advances.

Pros

  • One question at a time — clearest cognitive model.
  • Timing always asked when showTiming is on. Data contract preserved.

Cons

  • An extra click for every notePrompt question — slows the flow noticeably across a long category.
  • Reintroduces the papercut: rating buttons replaced by timing buttons, just one phase later.
  • Three states to remember; harder to "edit my last answer" cleanly.

Verdict: honest but the friction shows up across dozens of questions.

B-auto-now: default timing to "now"

Yes commits with timing: "now" automatically. The user fills a note and advances; no timing UI ever shown.

Pros

  • Identical UX to current (A) — no extra clicks, no extra UI.
  • Data contract technically preserved — timing field is set.

Cons

  • Encodes a guess into the journal. Wrong if the user meant "later" but never got asked.
  • Inflates "green-light" matches with answers the user didn't actually mark "now."
  • Misleading to the matching algorithm; misleading to the user reviewing their own results.
  • No way for the user to discover or correct the assumption.

Verdict: rejected — silent dishonesty about what the user said.
